## Critical Gaps

### Gap 1 — Field Name Divergence in `נרשמות` (CampaignEnrollments) Table

**Severity:** Critical — silently breaks enrollment display for contacts

**Location:** `src/lib/airtable/contacts.ts` lines 81, 88, 126, 131

**Issue:** `getContactEnrollments()` and `getEnrolledContactIds()` query the `נרשמות` table using field name `{אשת קשר}`. All other service files (`scheduler-services.ts`, `scheduled-messages.ts`) and the setup script comments use `איש קשר` as the canonical linked-record field name.

**Impact:**
- If Airtable field is `איש קשר` (canonical per setup script): `getContactEnrollments` returns empty arrays → contact detail panel shows no enrollment history (CONT-02 broken in production)
- `getContactMessages` (line 101) correctly uses `{איש קשר}` — so message history may still work
- Scheduler is unaffected (uses `איש קשר` correctly)

**Fix:** Verify actual Airtable field name in live base. If `איש קשר`, update lines 81, 88, 126, 131 in contacts.ts.

**Affected requirements:** CONT-02

---

### Gap 2 — Webhook Does Not Create Enrollment

**Severity:** Major — webhook contract documents a `campaign` parameter that is silently ignored

**Location:** `src/app/api/webhook/contact/route.ts` lines 46, 82–88

**Issue:** Route handler parses `body.campaign` (optional field per JSDoc) and validates the payload, but the Airtable create call only writes `שם מלא` and `טלפון`. No `CampaignEnrollment` record is created even when a campaign ID is provided.

**Impact:**
- Webhook-originated contacts are never auto-enrolled in a campaign
- A contact added via MAKE.com webhook with a campaign ID provided will appear in the contact list but not be counted as a campaign enrollee and will not receive scheduled messages
- To enroll such contacts, manual Airtable editing would be required

**Fix:** After creating the contact, if `body.campaign` is present, create a `נרשמות` record linking the new contact to the campaign.

**Affected requirements:** INFRA-04 (partial)
